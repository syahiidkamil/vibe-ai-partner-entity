# 2026-04-08: Multi-Language TTS Expansion + Chinese Model Integration

## Summary

Added 6 new languages (French, Spanish, Hindi, Italian, Portuguese, Mandarin Chinese) to the kokoro-onnx TTS engine, expanding from 3 languages / 13 voices to 9 languages / 34 voices. Encountered two significant issues during rollout.

- Added `ESPEAK_LANG_MAP` for espeak-based languages (French, Spanish, Hindi, Italian, Portuguese)
- Extended `VOICE_LANG_MAP` with new voice prefix mappings (`e`, `f`, `h`, `i`, `p`, `z`)

**File:** `packages/plugin-tts/kokoro-onnx/pyproject.toml`

- Added `misaki-fork[zh]` and `ordered-set` dependencies

## Issues Encountered

### Issue 1: Detached TTS Server Not Restarting (Stale Process)

**Symptom:** After code changes, `npm start` showed the server running but voices API returned old data (26 voices instead of 34).

**Root Cause:** The TTS server is spawned with `detached: true` and `tts.unref()` in `apps/cli/src/commands/start.ts`. When the user Ctrl+C's `npm start`, only the Node CLI and Vite dev server die. The TTS server survives as an orphaned process. On the next `npm start`, `checkHealth()` pings port 5111, finds the stale server healthy, and reuses it without spawning a new one.

**Relevant Code (`apps/cli/src/commands/start.ts:73-75`):**
```typescript
if (await checkHealth(ttsPort)) {
  await logEngineInfo(ttsPort);  // Reuses old server, no restart
} else {
  // Only spawns new server if health check fails
}
```

**Resolution:** Use `npm stop && npm start` to explicitly kill the old process before starting fresh. The stop command reads `.tts-server.pid` and sends SIGTERM.

**Lesson:** When developing/iterating on TTS engine code, always use `npm stop && npm start` — not just `npm start`. The health-check-first design is correct for production but creates a gotcha during development.

### Issue 2: Chinese Voice IDs Mismatch (v1.1-zh Model)

**Symptom:** Chinese TTS returned `{"status":"ok"}` but no audio played. No visible error because generation runs in a background `asyncio.create_task` that swallows exceptions.

**Root Cause:** The HuggingFace VOICES.md listed named Chinese voices (`zf_xiaobei`, `zf_xiaoni`, etc.) but the actual v1.1-zh model binary ships with **numbered voice IDs** (`zf_001`, `zf_002`, `zm_009`, etc.). The `kokoro.create()` call threw `AssertionError: Voice zf_xiaobei not found in available voices` which was silently caught by the async task.

**How it was found:** Ran generation directly in Python outside the server, which surfaced the assertion error. Then inspected `kokoro.voices.keys()` on the loaded zh model to discover the actual voice IDs.

**Resolution:** Updated voice entries to use the correct numbered IDs from the actual model binary.

**Lesson:** Always verify voice IDs against the loaded model (`kokoro.voices.keys()`) rather than trusting external documentation, which may reference a different model version.

### Architecture Note

Chinese uses a **separate ONNX model** (`kokoro-v1.1-zh.onnx` + `voices-v1.1-zh.bin` + `config.json`) that is lazy-loaded on first Chinese voice request. All other languages share the v1.0 model. The `_get_kokoro(voice)` method routes to the correct Kokoro instance based on voice prefix.

- `ordered-set` - Required by misaki Chinese module
